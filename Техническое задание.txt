

    1. ВВЕДЕНИЕ
Наименование программы – “Электронная библиотека”.
Программа разработана для хранения сведений об абонентах библиотеки и имеющейся литературе.


    2. НАЗНАЧЕНИЕ РАЗРАБОТКИ
Разработка данной программы ведётся в рамках курсового проекта.
В приложении будут реализованы следующие возможности:
    • Регистрация для нового пользователя и вход для зарегистрированного.
    • Все действия в системе производимые пользователем будут помещены в журнал.
    • Предусмотрена система ролей для управления доступом пользователей к функциям системы.
    • Возможность сделать запись на каждый экземпляр книги в формуляр с основными реквизитами, записями о выдаче читателю и возврате.
    • На каждого читателя заполняется абонементная карточка с перечнем полученных книг и ставится дата их возврата.


    3. ТРЕБОВАНИЯ К ПРОГРАММЕ
4.1. Требования к функциональным характеристикам.
4.1.1. Работа с базами данных.
На данные в базе должны накладываться следующие ограничения:
    • У каждого пользователя обязательно должны быть заполнены поля «Имя», «Фамилия», «Пароль», «Уровень доступа».
    • Максимальная длина фамилии: 30 символов, имени: 30 символов, пароля: 128 символов, уровня доступа: 1 символ.
    • Данные о книги содержат в себе название, автора, год выпуска, издательство, место выпуска, жанр и количество экземпляров.
    •  Максимальная длина названия книги: 4000 символов, автора книги, год выпуска: 4 символа, названия издательства: 100 символов,
     место выпуска книги: 100 символов, жанр книги: 50 символов, количество экземпляров: 1000 символов.
    • Данные об авторе содержат в себе имя автора: 100 символов, фамилию: 100 символов, отчество: 100 символов.
Интерфейс должен быть рассчитан на преимущественное использование «персональных компьютеров», т.е. управление системой должно
осуществляется с помощью набора экранных меню, кнопок, значков и т.п. элементов.
Пользователи должны взаимодействовать с системой на уровне графического пользовательского интерфейса.

4.2. Требования к нефункциональным характеристикам.
    • в качестве СУБД можно использовать любую версию MySQL,
    • количество таблиц в БД не менее 10,
    • доступ к данным осуществляется через слой хранимых процедур.
4.3. Уровни пользователей.
4.3.1. Пользователи.
Пользователи могут создавать и редактировать свою абонементную карточку, закреплять за собой книгу и удалять ее из своей карточки.
4.3.2. Администраторы.
Учетные записи разработчиков приложения хранятся в базе данных, по умолчанию данные пользователи являются администраторами.
Администраторы могут добавлять новые книги в базу данных, менять информацию о книгах, удалять книги из базы данных.
4.4. Требования к надежности.
Программный комплекс должен обеспечивать корректную обработку аварийных ситуаций, вызванных неверными действиями пользователей,
 неверным форматом или недопустимыми значениями входных данных. В указанных случаях система должна выдавать пользователю
 соответствующие аварийные сообщения, после чего возвращаться в рабочее состояние, предшествовавшее неверной (недопустимой)
  команде или некорректному вводу данных.
4.5. Требования к составу и параметрам технических средств.
В состав технических средств должен входить совместимый с IBM PC персональный компьютер.
	Для корректной работы сервера ПК пользователя должен иметь характеристики, удовлетворяющие системным требованиям используемого браузера.
4.6. Требования к информационной и программной совместимости.
4.6.1. Клиентская часть.
Пользователь может воспользоваться приложением с помощью следующих браузеров последней версии:
    • Google Chrome;
    • Яндекс.Браузер.

Браузеры для реализации интерактивных элементов клиентской части должны поддерживать JavaScript, отображение кириллицы
и латиницы, отображение изображений.
	  4.6.2. Серверная часть.
Используемые языки, программное обеспечение:
    • Windows 7, 8, 10; Linux; Windows Server (ОС);
    • Java (язык программирования);
    • MySQL. Можно другую
Для запуска сервера потребуется установить и настроить БД последней версии.
